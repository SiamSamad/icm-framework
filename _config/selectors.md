# Selector Quality Standard

This file defines the authoritative selector preference order for all Playwright tests generated by the ICM pipeline. Every stage that generates or validates selectors must follow this standard. Product configs (`_config/<product>.md`) list product-specific element selectors; this file defines the rules all of them must obey.

---

## Preference Order

Use the highest-ranked selector available for each element. Never skip a level without a documented reason.

### 1. `data-testid` (preferred)

Use `page.getByTestId('name')`.

**Why:** Completely immune to styling changes, copy changes, and DOM restructuring. Explicitly added by developers for testing ‚Äî signals intent and ownership.

### 2. Stable semantic `id`

Use `page.locator('#element-id')` when the `id` is clearly tied to the element's purpose (e.g. `#submit-button`, `#login-form`).

**Why:** Stable when hand-authored for a specific element. Avoid if the `id` looks auto-generated (e.g. `#input_237`, `#ember847`) ‚Äî treat those as brittle and move to level 3.

### 3. Role or label via `getByRole` / `getByLabel`

Use `page.getByRole('button', { name: 'Submit' })` or `page.getByLabel('Email address')`.

**Why:** Semantically meaningful, accessible, and naturally aligned with how users perceive the UI. Survives text and layout changes as long as the element's ARIA role and label remain consistent.

### 4. Structural CSS selector (last resort)

Use `page.locator('form > .submit-area button')` only when no higher-ranked option exists, and only when the selector targets DOM structure ‚Äî not visual styling classes.

**Why:** Acceptable for stable structural relationships (parent ‚Üí child, form context), but fragile when class names are generated or tied to design tokens. Never use utility classes like `.bg-blue-500`, `.text-sm`, or any class that could be renamed in a styling refactor.

### 5. Never ‚Äî XPath or brittle deep DOM chains

Do not use XPath (e.g. `//div[@class='...']`) or deep CSS chains (e.g. `div > section > ul > li:nth-child(3) > span`).

**Why:** Breaks on any markup change, carries no semantic meaning, and is impossible to maintain at scale. If no stable selector exists at levels 1‚Äì4, add a `data-testid` to the component instead and flag it in the Stage 02 testability report.

---

## When No Stable Selector Exists

If levels 1‚Äì4 all fail for an element:

1. Generate the test with a clearly marked placeholder:
   ```typescript
   // TODO: selector missing ‚Äî add data-testid="<suggested-name>" to this element
   page.locator('PLACEHOLDER')
   ```
2. Record the gap in the Stage 02 testability report under "elements missing a stable selector".
3. Never fall back to XPath or deep chains as a shortcut.

## Quick Reference

| Rank | Method | Use when |
|------|--------|----------|
| 1 | `getByTestId('name')` | `data-testid` attribute exists |
| 2 | `locator('#id')` | Hand-authored semantic id exists |
| 3 | `getByRole` / `getByLabel` | Accessible element with stable role/label |
| 4 | Structural CSS | No semantic anchor; target structure not style |
| ‚úó | XPath / deep chains | Never |

**Note for page object code (Stage 04):** When writing TypeScript page objects, do not use CSS class selectors at all ‚Äî not even structural ones. The preference order in generated code is: `getByTestId` ‚Üí `getByRole` ‚Üí `getByLabel` ‚Üí `getByText`. If none of these work, add a `data-testid` to the component rather than falling back to CSS.

## Three-Path Testability Strategy (Stage 02)

When classifying whether a UI element can be reliably targeted, check sources in this order:

1. **Product config selector table** (`_config/<product>.md`) ‚Äî if a `data-testid` or preferred selector is listed there, the element is selectable. Use it.
2. **Stage 01 code diff** (if `sources_read.code_diff.status: read`) ‚Äî look for `data-testid`, stable `id`, or ARIA attributes in the diff. Follow the preference order above.
3. **Classify based on what was found:**
   - Selector confirmed ‚Üí **Elements with a usable selector**
   - Diff was read (`status: read`) but no stable selector found ‚Üí **Elements missing a usable selector** (flag gap, recommend `data-testid`)
   - Diff not available (`status: not_found` or `unavailable`) ‚Üí **Testability unverified** (selector may exist; we couldn't check ‚Äî do not flag as missing)

## `data-testid` Naming Conventions

Recommended values must match the naming style already used in the product config. Use the prefix that fits the element type:

| Element type | Prefix | Example |
|-------------|--------|---------|
| Button | `btn-` | `btn-confirm-pickup` |
| Modal / dialog | `modal-` | `modal-location-permission` |
| Banner / alert | `banner-` | `banner-error` |
| Input / field | `input-` | `input-carrier-name` |

Do not invent arbitrary or inconsistent names. When recommending a new `data-testid`, match the style of existing values in `_config/<product>.md`.
